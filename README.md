# Keytie — защищённый мессенджер со сквозным шифрованием

Курсовой проект по дисциплине «Методы и технологии программирования», вариант 5: реализация протокола безопасного обмена сообщениями с end-to-end шифрованием на основе протокола Signal (X3DH + Double Ratchet).

Сообщения шифруются на устройстве отправителя и расшифровываются только на устройстве получателя; сервер выступает «тонким» ретранслятором и не имеет доступа к содержимому переписки.

## Возможности

- Сквозное шифрование на основе протокола Signal (рукопожатие X3DH и алгоритм Double Ratchet).
- Совершенная прямая секретность (PFS) и устойчивость к компрометации.
- Диалоговая и групповая переписка (группы реализованы попарной рассылкой с сохранением PFS для каждого получателя).
- «Тонкий» сервер-ретранслятор, оперирующий только зашифрованными данными.
- Графический клиент на PyQt6.

## Архитектура и технологический стек

- **Криптографическое ядро** — Rust (примитивы X25519, ChaCha20-Poly1305, HKDF, HMAC-SHA256), подключается к Python через PyO3 и собирается инструментом maturin.
- **Клиент** — Python: логика протокола (X3DH, Double Ratchet) и графический интерфейс на PyQt6.
- **Сервер** — FastAPI с хранилищем SQLite, выполняет роль ретранслятора сообщений.

## Требования

- Python 3.11
- Rust (вместе с cargo) и maturin для сборки криптографического ядра
- Docker и Docker Compose (опционально, для запуска сервера в контейнере)

## Установка и запуск

1. Клонировать репозиторий и перейти в каталог проекта:
   ```
   git clone https://github.com/ninjelle/Keytie-messenger.git
   cd Keytie-messenger/secure-messenger
   ```
2. Создать и активировать виртуальное окружение:
   ```
   python -m venv .venv
   .venv\Scripts\activate          (Windows)
   source .venv/bin/activate        (Linux/macOS)
   ```
3. Установить зависимости Python:
   ```
   pip install -r requirements.txt
   ```
4. Собрать криптографическое ядро на Rust:
   ```
   cd secure_core
   maturin develop
   cd ..
   ```
5. Запустить сервер-ретранслятор одним из способов:
   - в контейнере Docker (из каталога `secure-messenger`):
     ```
     docker compose up --build
     ```
   - локально:
     ```
     cd server
     uvicorn main:app --host 0.0.0.0 --port 8000
     ```
6. Запустить клиент (для нескольких собеседников — в отдельных терминалах):
   ```
   cd client
   python gui.py
   ```

Для диалога в окне клиента указывается имя собеседника. Для группового чата в поле собеседника перечисляются имена участников через запятую; все участники должны подключиться, после чего можно начинать переписку.

Примечание для Windows: при ошибке кодировки во время сборки ядра выполнить `set PYTHONUTF8=1` перед командой `maturin develop`.

## Тестирование

Автоматические тесты запускаются фреймворком pytest из каталога `client`:
```
cd client
pytest -v
```
Тесты покрывают криптографические примитивы, рукопожатие X3DH, алгоритм Double Ratchet (включая проверку прямой секретности) и групповую переписку.

## Безопасность

В проекте применяются статический анализ кода и анализ зависимостей на уязвимости:
- `bandit` — статический анализ Python-кода (SAST);
- `pip-audit` и `cargo audit` — проверка зависимостей Python и Rust на известные уязвимости (SCA).

Все проверки, наряду с тестами, автоматически выполняются в конвейере непрерывной интеграции (GitHub Actions) при каждом изменении кода.

## Документация

Проект сопровождается следующей документацией:
- **Настоящий README** — общее описание, требования и инструкция по запуску.

- **Диаграммы** (каталог `docs/`) — UML-диаграммы компонентов, последовательности и классов, иллюстрирующие архитектуру и взаимодействие модулей.

## Использование ИИ-инструментов при разработке

В соответствии с требованиями дисциплины при работе над проектом применялись ИИ-ассистенты — в роли вспомогательного инструмента, близкого к интерактивному справочнику и наставнику. С их помощью разбирались теоретические основы протокола Signal, уточнялись детали криптографических стандартов (RFC), готовились черновые фрагменты шаблонного кода, пояснялись сообщения об ошибках при отладке, а также давались рекомендации по инструментам анализа безопасности.

Вместе с тем проектные решения — выбор архитектуры, способа реализации групповых чатов, структуры модулей — принимались автором; код интегрировался, проверялся и тестировался самостоятельно, а предлагаемые ИИ варианты критически оценивались и адаптировались под задачи проекта. Таким образом, ИИ-инструменты ускорили разработку и помогли глубже разобраться в предметной области, но не заменили инженерных решений и понимания, лежащих в основе работы.
